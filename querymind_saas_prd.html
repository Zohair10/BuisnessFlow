What Is This Product?

QueryMind is a multi-tenant SaaS platform that lets non-technical business users talk to their databases in plain English. The AI converts natural language questions into SQL or NoSQL queries, executes them against connected data sources (PostgreSQL, MongoDB, CSV uploads), and intelligently decides whether to respond with a text summary, a data table, or a chart — whichever best communicates the answer.

Core User Problem

😤 Without QueryMind
Business users wait for developers to write reports. Data analysts spend days building dashboards. Decisions are made on stale data because querying is too hard for non-engineers.
✅ With QueryMind
A sales manager types "show me top 10 customers by revenue this month" and gets a chart in 3 seconds — no SQL, no dashboards, no tickets to engineering.

Target Personas

📊 Business Analyst
Wants fast answers from data without writing SQL. Uses QueryMind daily for ad-hoc reporting.
🏪 SMB Owner
Connects their Shopify / inventory DB. Asks questions like "what's my best selling item this week?"
👩‍💻 Technical Lead
Sets up the DB connections & manages team members. Cares about security, multi-source joins.

Non-Goals (v1)

✗
No write operations — QueryMind is read-only. No INSERT, UPDATE, DELETE via AI chat.
✗
No real-time streaming — responses are request-response, not live data streams in v1.
✗
No custom dashboard builder — charts are inline in chat only. Pinning to dashboards is a v2 feature.
✗
No on-premise deployment — cloud SaaS only (Vercel + Railway or AWS) in v1.

Key Architectural Decisions

🔐 DB Credential Isolation
Each workspace's database credentials are stored encrypted in QueryMind's platform DB. Decryption happens only inside the Node.js API layer. Credentials never leave the server — frontend never sees them.
🛡️ Read-Only Query Enforcement
Every AI-generated SQL query runs through a sanitizer that rejects any non-SELECT statement. The DB connection itself is created with a read-only role when possible.
📊 Two Claude Calls Per Query
Call 1: NL → SQL (given schema context). Call 2: raw rows → format decision + human summary. Both are fast models; M1 uses one call, M2 adds the second.
🗄️ Schema Introspection Cache
On connection setup, table schemas are extracted and stored. This schema is injected into the Claude system prompt so it knows table names, column names, and types — crucial for accurate SQL.

Multi-Tenant Isolation Strategy

T01
Workspace-based tenancy
Every resource (DB connections, query history, API keys) is scoped to a workspace_id. Cross-workspace access is impossible at the query level.
T02
Row-level security on platform DB
Postgres RLS policies ensure API routes can only read/write their own workspace's rows, even if there's a bug in the application logic.
T03
T03
Rate limiting per workspace
Redis-based rate limiting caps LLM calls per workspace per minute to prevent abuse and cost explosion.

F1.4

User question + schema context → Claude generates SQL. The generated SQL is shown to the user (collapsible) for transparency.

User uploads a CSV or Excel file → system parses it and loads it into an in-memory SQLite table → Claude can query it exactly like a relational DB.

Core Concept: The Format Decision Loop

After SQL execution returns raw rows, a second Claude call analyzes the data shape and question intent to decide the best format. This is the key differentiator over a basic "show me a table" tool.
Raw Rows (JSON)
→
Claude: Format Decision
→
{ type, config, summary }

Format Decision Rules (Claude System Prompt)

const formatPrompt = ` Given this data and the original question, return JSON: { "type": "text" | "table" | "bar_chart" | "line_chart" | "pie_chart", "chartConfig": { xKey, yKey, title, color } or null, "summary": "Human-readable answer in 1-2 sentences" } Decision rules: - Single number/fact → "text" - < 5 rows with > 3 columns → "table" - Comparing categories (top N, by region...) → "bar_chart" - Data over time (monthly, weekly, daily) → "line_chart" - Part-of-whole (percentages, distribution) → "pie_chart" - > 20 rows with no clear chart axis → "table" `

Functional Requirements

F2.1
Format Decision Engine
Second Claude call receives raw data rows + original question → returns structured JSON with type + chart config + summary text.
F2.2
Dynamic Chart Renderer
Frontend receives chartConfig and renders the correct Recharts component (BarChart, LineChart, PieChart). All charts are fully labeled with axes and tooltips.
F2.3
Interactive Data Table
Table responses have sortable columns, row count display, and a CSV download button.
F2.4
Text Response with Markdown
Single-value answers are shown as a styled insight card (e.g. "Total revenue: $1.2M") with supporting context from Claude.
F2.5
User Format Override
Users can manually switch between Table / Chart / Text view after the AI decides. Format preference is remembered per query type.
F2.6
Chart Export
Any chart can be exported as PNG. Tables export as CSV. Insight cards copy to clipboard.

MILESTONE 03 — FUTURE PLANNING
Inventory Alerts & Background Jobs
User-defined rules → background monitoring → notifications
Priority: LATER
⚠️
M3 is defined here for planning purposes but should NOT be built until M1 and M2 are stable and have real users. The background job infrastructure is a significant architectural addition.

Core Concept

Users define rules like "alert me when stock_quantity < 10 for any product" — and a background worker periodically runs the corresponding SQL query and fires a notification if the condition is met.

New Infrastructure Required for M3

🔄 Job Queue
BullMQ (Redis-backed) — schedule recurring alert check jobs per rule. Workers run on Railway, separate from the Next.js server.
⏰ Cron Scheduler
Each alert rule has a schedule (every 5 min, hourly, daily). BullMQ handles the cron. No external cron service needed.
📬 Notification Delivery
Email via Resend, Slack webhook, and in-app notification bell. User picks delivery method per alert.
📊 Alert History
Every alert trigger is logged with: rule, condition met, timestamp, data snapshot that triggered it.

SaaS-Specific Concerns

The things that separate a "works for me" project from a real product you can sell.

Security Checklist

SEC1
Credential Encryption at Rest
DB passwords and connection strings → AES-256-GCM encrypted before storing in PostgreSQL. Decryption key in environment variables, never in code.
SEC2
Read-Only DB Access
Advise users to create a read-only DB user for QueryMind. Enforce SELECT-only at the application level as a second layer.
SEC3
IDOR Prevention
Every API route validates that the requested connectionId / queryId belongs to the authenticated user's workspace. Never trust client-supplied IDs without DB ownership check.
SEC4
Prompt Injection Defense
User questions are injected into Claude prompts. Strip/escape control sequences. System prompt explicitly instructs Claude to only generate SQL and ignore instructions in the user message.
SEC5
Data Residency Disclosure
Query results (row data) are sent to Claude's API for summarization. Be upfront in privacy policy. Offer an "on-premise" tier in the future for enterprise customers with sensitive data.

Key LLM Cost Controls

🔄 Query Result Caching
Same question + same connection + same data hash → return cached response. Skips both Claude calls. Use Redis with 5-minute TTL for volatile data.
✂️ Row Truncation
Only send first 100 rows to Claude for summarization/format decision. SQL can return 1000 rows to the user, but Claude only processes a sample.
💸 Cost Per Query Tracking
Log token usage per Claude call. Track cost per workspace per month. Alert if a workspace's LLM cost approaches 80% of their subscription revenue.
📉 Rate Limiting
Hard cap of 10 queries/minute per workspace via Redis sliding window. Soft cap with a warning at 80% of monthly limit to encourage upgrades.
